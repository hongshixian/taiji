# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

太极 (Taiji) — Flask 3 + Vue 3 + Celery + Redis full-stack scaffold. Ships with JWT auth (RBAC + JWT revocation), multi-tenant data isolation (shared schema, global users + membership-based tenant access), Celery-backed example tasks (webpage analysis and CSV data quality checks), rate limiting via flask-limiter, and a one-shot `docker compose up` deploy. SQLite by default; switch via `DATABASE_URL`.

## Common commands

```bash
# ── Backend dev (from backend/) ──
pip install -r requirements.txt
flask db upgrade                                   # apply migrations
python run.py                                      # Flask on :5000

# ── Celery worker (separate terminal, from backend/) ──
celery -A celery_app worker -l info                # uses module-level `app` in celery_app.py

# ── Frontend dev (from frontend/) ──
npm install
npm run dev                                        # Vite on :5173, proxies /api -> :5000
npm run build                                      # production bundle

# ── Tests (from backend/) ──
conda run -n py12pt python -m pytest tests/ -v                         # all tests
conda run -n py12pt python -m pytest tests/test_auth.py -v             # one file
conda run -n py12pt python -m pytest tests/test_auth.py::test_login -v # one test
make test                                          # repo-root shortcut

# ── Database migrations (from backend/) ──
flask db migrate -m "describe change"              # generate migration from model diff
flask db upgrade                                   # apply
flask db downgrade                                 # roll back

# ── Docker (from repo root) ──
docker compose up -d                               # full stack on :80
docker compose down
make build / make up / make down
```

Default admin: seeded by `docker/entrypoint.sh` on **first** start (skipped when an admin already exists). If `ADMIN_PASSWORD` env var is set, it's used; otherwise a random password is generated (`secrets.token_urlsafe(12)`, ~16 chars) and printed to stdout once — read it with `docker compose logs backend | grep -A 5 "首次部署"`. Username/email come from `ADMIN_USERNAME` / `ADMIN_EMAIL` (default `admin` / `admin@taiji.local`). The seeded admin is created with `is_superuser=True` and added to the `default` tenant as owner. **There is no hardcoded default password anymore** — old setups still using `admin/admin123` continue to work because seed is skipped when an admin row exists.

## Architecture

### Backend (`backend/`)

- **App factory** (`app/__init__.py`): `create_app(config_obj=Config)` wires SQLAlchemy, Flask-Migrate, JWT, flask-limiter, CORS, and Celery. Models are imported eagerly inside `create_app` so SQLAlchemy sees them before `init_app`. **All business blueprints are mounted under `/api/v1/`** (constant `API_V1` in `__init__.py`): `/api/v1/auth`, `/api/v1/tasks`, `/api/v1/tasks/webpage-analysis`, `/api/v1/tasks/csv-quality`, `/api/v1/admin`, `/api/v1/superadmin`, `/api/v1/audit-logs`. `/api/health` lives directly on the app outside the version prefix (stable across API versions for monitoring/probes). A `@before_request` hook reads JWT claims into `g.tenant_id` / `g.is_superuser` / `g.bypass_tenant_filter` for the tenant filter machinery (see below). **Production environment rejects insecure default secrets** — `Config._check_secrets()` raises `RuntimeError` if `SECRET_KEY` or `JWT_SECRET_KEY` uses a well-known default value when `FLASK_ENV=production`.
- **Multi-tenancy** (`app/models/_tenant_mixin.py` + `app/models/tenant_membership.py`): `users` is the global login identity (`username` / `email` are globally unique — no `tenant_id` on the User model). Tenant membership lives in `tenant_memberships` (`user_id`, `tenant_id`, `role_id`, `is_active`, `is_owner`), so one user can belong to multiple tenants and hold different roles per tenant. Business data models still inherit `TenantMixin`, which adds `tenant_id` and globally injects `WHERE tenant_id = g.tenant_id` via `with_loader_criteria`. `User` does **not** inherit `TenantMixin`; `TenantMembership` does. Use `bypass_tenant_filter()` only for public auth flows, tenant membership resolution, superadmin stats, CLI/Celery-style cross-tenant work, or migrations. Tenant has only identity/status/protection fields (`slug`, `name`, `is_active`, `is_system`) — no plan/billing field. System tenants `default` (slug constant `DEFAULT_TENANT_SLUG`) and `guest` (slug constant `GUEST_TENANT_SLUG`, both in `app/models/tenant.py`) are seeded via migration + entrypoint fallback.
- **RBAC** (`app/permissions.py` + `app/models/role.py`): permissions are hardcoded enum values (`Permission.USER_READ` etc.) seeded into `permissions`; roles bind permissions via `role_permissions`. System roles `admin` / `user` / `guest` are global (`tenant_id=NULL`) and immutable; custom roles are tenant-scoped (`roles.tenant_id`) so different tenants can define the same role name with different permissions. Role lookup resolves tenant custom roles first, then system roles. `tenant_memberships.role_id` is the authoritative role link. **Use `@require_permission(Permission.XXX)` on new handlers**; the decorator reads `perms` from the current JWT claim, which is derived from the active membership. There is intentionally no `@admin_required` shortcut; express admin capabilities through explicit permission codes or `@superuser_required`.
- **Superuser** (`users.is_superuser`): platform operators can manage tenants through `/api/v1/superadmin/*`, gated by `@superuser_required`. Superuser **no longer** automatically bypasses tenant filtering on normal business endpoints; normal endpoints remain scoped to the current JWT `tenant_id`. Creating a tenant via superadmin auto-adds the creator as an admin+owner membership for that tenant.
- **System settings** (`app/models/system_setting.py` + `app/services/system_setting_service.py` + `/api/v1/superadmin/settings`): platform-level key/value JSON settings, not tenant-scoped, only available to `is_superuser` users. Settings are schema-gated: only keys defined in `SETTING_DEFINITIONS` (in `system_setting_service.py`) are accepted. The first setting is `public.default_registration_tenant_slug`, which controls which tenant new public registrations join (defaults to `"guest"` via `GUEST_TENANT_SLUG`).
- **Audit logs** (`app/models/audit_log.py` + `app/services/audit_log_service.py` + `/api/v1/audit-logs`): append-only DB records for platform management actions, separate from task execution logs. `AuditLog` intentionally does **not** inherit `TenantMixin`; query scope is enforced in `list_audit_logs()` so superusers can query across tenants while tenant admins only see their current tenant. Service-layer mutations call `record_audit_log()` in the same transaction; do not use a generic request hook to infer audit actions. `record_audit_log()` reads the actor (user_id, username, is_superuser) and `tenant_id` from the JWT / `g` automatically — callers only pass `action`, `resource_type/id/name`, and optional `before_data`/`after_data`/`metadata`. **Do not commit inside `record_audit_log()`** — it only `db.session.add()`s so the audit row shares the calling business transaction.
- **Superadmin management APIs** (`app/api/superadmin.py`): all gated by `@superuser_required`:
  - Tenant CRUD: `GET/POST /tenants`, `GET/PUT/DELETE /tenants/<id>` — creating a tenant auto-adds the creator as owner.
  - Tenant member management: `GET/POST /tenants/<id>/members`, `DELETE /tenants/<id>/members/<user_id>` — adds existing global users by username/email; user creation stays in registration/user-management flows.
  - Superuser management: `GET/POST /superusers`, `DELETE /superusers/<id>` — cannot remove own superuser status.
  - System settings: `GET/PUT /settings`.
  - Helper: `GET /roles?tenant_id=<id>` — lists system roles plus that tenant's custom roles for superadmin UI.
  - Tenant switch: `POST /switch-tenant` — superadmin variant (still requires an active membership in the target tenant).
- **User identity payload** (`auth_service.user_to_dict` + `/api/v1/auth/me`): `/auth/me` returns the global user, `current_tenant` (object with `id`/`slug`/`name`), current membership `role`/`role_name`/`permissions`/`membership_id`/`membership_active`, and `memberships` array for all joined tenants. JWT additional claims carry `tenant_id`, `membership_id`, `perms`, and `is_superuser`. `/api/v1/auth/switch-tenant` validates that the user has an active membership in the target tenant (and the tenant is active), then issues a new access token for that tenant.
- **Registration flow** (`auth_service.register_user`): creates a global user then auto-adds a `user`-role membership in the tenant determined by `system_setting_service.get_default_registration_tenant_slug()` (defaults to `"guest"`). The register endpoint does **not** accept `tenant_slug` from the client; tenant assignment is server-controlled via system settings.
- **Login flow** (`auth_service.login_user`): accepts only global `username` + `password`, selects the user's first active membership where the tenant is also active (via `_select_login_membership()`), and returns tokens scoped to that membership.
- **JWT revocation** (`app/utils/jwt_blocklist.py` + `users.tokens_revoked_at`): two revocation channels. (1) Single-token revoke via Redis blocklist on `jti` — used by `POST /auth/logout`; fail-open on Redis outage. (2) User-wide revoke via `tokens_revoked_at` timestamp — `update_user` sets it whenever password / role / membership_active / is_active changes, and `POST /auth/password` (self-service password change), superuser changes, and tenant member changes do the same. The `@jwt.token_in_blocklist_loader` checks both. The revoke marker is rounded **up to the next second** (`_revoke_marker()` in `auth_service.py`) so a fresh login immediately after revocation doesn't get caught by the same-second iat collision.
- **Task architecture** (`app/models/task.py` + business detail tables): `tasks` stores lifecycle fields (`tenant_id`, `user_id`, `task_type`, `status`, errors, timestamps) plus `log_path` for optional task execution logs. Each business owns a separate detail table and module, e.g. `webpage_analysis_tasks` with `webpage_analysis_service.py`, and `csv_quality_tasks` with `csv_quality_service.py`. Do not add business-specific nullable fields to `tasks`. `Task.user_id` has `ondelete="CASCADE"` — deleting a user cascades to their tasks.
- **Task logs** (`app/services/task_log_service.py` + `GET /api/v1/tasks/<id>/logs`): task execution logs are JSONL files, not DB rows. `create_task_record()` assigns `tasks.log_path` as `tasks/tenant_{tenant_id}/{task_type}/task_{task_id}.jsonl`; backend and worker share `TASK_LOG_ROOT` (`/app/logs` in Docker Compose, `../app_logs` locally). Business tasks may call `create_task_logger(task).info/warn/error(...)` but can also skip task logs entirely. Read APIs must first resolve the `Task` through tenant-scoped DB queries, then read the file.
- **Celery wiring** (`celery_app.py`): module-level `celery = Celery("taiji")` is created at import time, then `create_celery_app()` runs `create_app()` and calls `init_celery(flask_app)` to copy Flask config into Celery and bind `celery.flask_app`. The Worker entry point is `celery -A celery_app worker` — it picks up the module-level `app = create_celery_app()`. **All Celery task DB work must run inside `with celery.flask_app.app_context():`**. **Tasks must explicitly carry `tenant_id`** as an argument since worker processes don't inherit request context; use `bypass_tenant_filter()` inside the task to read the original record.
- **Layering**: `api/` (blueprints) → `services/` (business logic, DB writes) → `models/` (SQLAlchemy). Request validation uses **marshmallow** schemas in `app/schemas/` invoked via `validate_schema()` helper (`app/utils/validation.py`) which returns `(parsed, None)` on success or `(None, (response, status))` on failure — handlers must check `error` and early-return.
- **SSRF protection** (`app/utils/ssrf.py`): `validate_url()` rejects private/reserved IPs, internal hostnames (localhost, *.local, *.internal, host.docker.internal), and non-http(s) schemes. DNS resolution is checked before the request is made. `safe_requests_get()` wraps `requests.get()` with SSRF validation and manual redirect following (each redirect target is re-validated). The webpage analysis schema (`WebpageAnalysisSubmitSchema`) validates URLs at submit time; the worker also validates before fetching.
- **Auth & roles**: JWT identity is stored as a **string** (`create_access_token(identity=str(user.id))`) — always `int(get_jwt_identity())` when reading. JWT additional_claims carry `perms` (list of permission codes), `tenant_id`, `membership_id`, and `is_superuser`. **JWT carries enough data that the permission decorator never queries the DB** — membership/role changes must revoke tokens so stale claims are invalidated.
- **Rate limiting**: shared `limiter` from `app/__init__.py` is used by task APIs such as `webpage_analysis_bp` and `csv_quality_bp` via `@limiter.limit(...)`. The auth blueprint creates its **own** `Limiter` instance (`auth_limiter` in `app/api/auth.py`) for register (5/min), login (10/min), refresh (30/min), password change (5/min), and tenant switch (10/min) — this is intentional, keeping auth limits independent from other endpoints. Both limiters use `_get_client_ip()` from `app/__init__.py` which reads `X-Forwarded-For` / `X-Real-IP` headers for correct rate limiting behind reverse proxies. The admin blueprint also rate-limits user creation (20/min) and role creation (20/min). 429 responses are normalised by the app-level errorhandler in `create_app()`.
- **Response envelope**: every JSON response follows `{"code": int, "message": str, "data": ...}` — `code: 0` means success, non-zero is a **business error code** from `app/utils/errors.py::ErrorCode` (not the HTTP status). Stick to this when adding endpoints.
- **Response helpers** (`app/utils/response.py`): handlers should return via `ok(data, message)`, `created(data, message)`, or `paginated(items, total, page, per_page)` — **never hand-roll `jsonify({"code": 0, ...})`** in handlers. The `paginated` helper guarantees the `items/total/page/per_page/pages` shape.
- **Business errors** (`app/utils/errors.py`): raise `BusinessError(ErrorCode.XXX)` from services and decorators; a global errorhandler converts it to the envelope with the right HTTP status. Codes are segmented: `1xxxx` user/auth, `11xxx` tenant, `2xxxx` tasks, `3xxxx` JWT/permissions, `9xxxx` system. Add new codes to `ErrorCode` rather than passing ad-hoc strings.
- **Config** (`config.py`): `Config` (production-ish) and `TestConfig` (in-memory SQLite + `CELERY_TASK_ALWAYS_EAGER = True`). `.env` is loaded by `python-dotenv` at import time. Production startup rejects insecure default secrets via `Config._check_secrets()`. `CORS_ORIGINS` env var (comma-separated) controls allowed origins; defaults to `["http://localhost:5173", "http://127.0.0.1:5173"]` in dev.
- **Migrations**: managed via Flask-Migrate (Alembic). Migration files live in `backend/migrations/versions/`. This demo baseline is currently compressed into a single initial migration (`0001_initial_schema.py`); do not keep one-off historical data migration code unless the project has real deployed data to preserve. Always generate a migration when models change — do not rely on `db.create_all()` outside tests.

### Frontend (`frontend/`)

- **Vue 3 + Vite + Pinia + Element Plus**, hash-mode router. Views: `Home`, `Login`, `Register`, `TaskManagement` (webpage analysis), `CsvQualityManagement`, `UserManagement`, `RoleManagement`, `TenantManagement`, `AuditLogManagement`, `SystemSettings`, `Settings`. Reusable widgets live under `src/components/` (`TenantSwitcher.vue` in the App header, `TaskLogDialog.vue` for execution logs).
- **`src/api/request.js`** is the only HTTP entry point. It auto-attaches `Bearer <accessToken>` from `localStorage`, and on 401 it transparently calls `/api/v1/auth/refresh` with the refresh token, queues concurrent failed requests, then replays them. If refresh fails, it calls `authStore.logout()` and redirects to `#/login` (hash-mode routing — **never use `window.location.href = '/login'`** as it breaks hash routing). Any new API module under `src/api/` should import this instance, not raw axios. **baseURL is `/api/v1`** — API modules use relative paths like `/auth/login`.
- **Auth store** (`src/stores/auth.js`): `user` holds the `/me` payload (`is_superuser`, `current_tenant`, current `role`/`role_name`/`permissions`/`membership_id`/`membership_active`, and `memberships`). Computed: `isSuperuser`, `currentTenant` (id/slug/name), `originalTenantId`. Actions: `switchTenant(tenantId)` calls `/auth/switch-tenant`, swaps `accessToken`, then re-fetches `/me`; `resetTenant()` switches back to the login tenant via `originalTenantId`. `originalTenantId` is persisted in localStorage for "切回登录租户".
- **Frontend API modules**: `src/api/auth.js` (register, login, refreshToken, getMe, listMyTenants, switchTenant, logout, changePassword), `src/api/admin.js` (user/role CRUD), `src/api/superadmin.js` (tenant CRUD, system settings, superuser management, tenant member management, roles helper, switch-tenant), `src/api/audit.js` (audit log listing), `src/api/task.js` (generic task listing + task logs), `src/api/webpageAnalysis.js`, and `src/api/csvQuality.js` (per-task-type submit/list/retry/delete). New API modules should follow the same pattern — import `request` from `./request`, never use raw axios.
- **Router guards** (`src/router/index.js`): `meta.requiresAuth` / `meta.guest` / `meta.requiresPermission` (single permission code) / `meta.requiresSuperuser`. On first navigation with a token but no in-memory user, it lazy-fetches `authStore.fetchUser()`. Permission failures redirect to `/`.
- **Routes**: `/` (Home), `/login` (guest), `/register` (guest), `/tasks` (auth + redirect), `/tasks/webpage-analysis` (auth + `task:read`), `/tasks/csv-quality` (auth + `task:read`), `/users` (auth + `user:read`), `/roles` (auth + `role:read`), `/audit-logs` (auth + `system:audit`), `/tenants` (auth + superuser), `/settings` (auth), `/system-settings` (auth + superuser).
- **Tenant switcher** (`src/components/TenantSwitcher.vue`, mounted in App.vue header): renders a dropdown when the current user has more than one active membership; otherwise it renders a read-only current-tenant tag. Switching calls `authStore.switchTenant()` then `window.location.reload()` so page data refetches under the new tenant.
- **System settings page** (`src/views/SystemSettings.vue`): mounted at `/system-settings`, guarded by `meta.requiresSuperuser`. Contains: default registration tenant selector, superuser list management (add/remove, cannot remove self). Separate from personal/general settings at `/settings`.
- **Task type navigation** (`src/constants/taskTypes.js` + sidebar in `App.vue`): top-level "任务管理" is an expandable menu group, not a page. Task-type routes live under `/tasks/...`; the current first type is `webpage_content_analysis` at `/tasks/webpage-analysis`. Add future task types by extending `TASK_TYPE_ROUTES`, adding a child menu item, and creating a dedicated API/module for that task type.
- **Webpage analysis page** (`src/views/TaskManagement.vue`): mounted at `/tasks/webpage-analysis`. It keeps newly submitted/retried jobs in a local `activeTasks` polling list, then moves completed/failed jobs into the backend-backed history table. History loads via `listWebpageAnalyses(page, perPage)` and is scoped to the current tenant. **Active task mutations must go through the reactive proxy** (`_updateActiveTask(taskId, updates)` / `_getActiveTask(taskId)`) rather than modifying the raw closure object directly — otherwise Vue reactivity won't trigger re-renders for elapsed timers and status changes.
- **CSV quality page** (`src/views/CsvQualityManagement.vue`): mounted at `/tasks/csv-quality`, submits a task name plus a local `.csv` file via multipart upload, then renders row/column counts, empty values, duplicate rows, inferred types, preview rows, and warnings. Same reactive proxy pattern as TaskManagement for active task polling.
- **Task log UI** (`src/components/TaskLogDialog.vue`): task pages use the shared dialog to read `/tasks/<id>/logs`. In task tables, keep the log entry after the status column and before destructive/retry operations; active task cards show the log button next to the status once the backend task id exists.
- **Tenant management page** (`src/views/TenantManagement.vue`): mounted at `/tenants`, guarded by `meta.requiresSuperuser`. Superusers can: list all tenants with user/task stats, create/edit/delete tenants, switch only to tenants where they have an active membership, and manage tenant members (add existing global users by username/email). System tenants are protected from deletion. Validates slug format (lowercase, numbers, hyphens); there is no plan selector.
- **Audit log page** (`src/views/AuditLogManagement.vue`): mounted at `/audit-logs`, guarded by `meta.requiresPermission: 'system:audit'`. Filters by tenant (superuser only), actor, action, resource type/id, result, and date range. Tenant admins see only their current tenant; superusers see all and can filter by tenant.
- **Theme system** (`src/assets/theme.css`): plain-CSS variables (no SCSS). Brand palette is ink-black + cinnabar + jade + gold, exposed as `--taiji-primary` / `--taiji-accent` / `--taiji-jade` / `--taiji-gold` and the gradients `--taiji-gradient-{brand,accent,hero}`. The file also overrides Element Plus tokens (`--el-color-primary`, radii, etc.) and defines shadows/radii. **Style new pages with these variables, not hard-coded colors** — they auto-flip under `html.dark`.
- **Dark mode**: toggled by adding/removing `html.dark`; persisted in `localStorage['taiji-theme']`. Initialized in `main.js` (respects `prefers-color-scheme` when no saved value). The user-facing switch lives in the App.vue header and Settings page.
- **Icons**: `@element-plus/icons-vue` is registered globally in `main.js` (`for (const [name, comp] of Object.entries(...)) app.component(name, comp)`) — use `<el-icon><HomeFilled /></el-icon>` directly, no per-file import. **Do not use emoji for UI icons.**
- **Route transitions**: `<transition name="fade-slide">` wraps `<router-view>` in App.vue; the keyframes live in `theme.css`.
- **Dev proxy**: `vite.config.js` proxies `/api` → `http://localhost:5000`. In Docker, Nginx (`docker/nginx.conf`) proxies `/api/` → `backend:5000` and falls back unknown paths to `index.html` for SPA routing.

### Deployment (`docker/`)

Four services: `redis`, `backend` (gunicorn 4 workers via `docker/entrypoint.sh` — runs `flask db upgrade` then conditionally seeds an admin: uses `ADMIN_USERNAME/EMAIL/PASSWORD` env vars if set, else generates a random password and prints it once; skipped entirely when an admin row already exists. Also ensures system tenants `default` and `guest` exist as a fallback. The seeded admin gets `is_superuser=True` and is added to the `default` tenant as owner), `worker` (same image, `celery worker` command, `RUN_MIGRATIONS=false` so only backend applies migrations), `frontend` (multi-stage build → nginx). SQLite DB persists in `./app_data:/app/data` on the host. `backend` and `worker` both wait for redis `healthcheck` to pass.

Task execution logs persist in `./app_logs:/app/logs`; both backend and worker mount this same path so worker can append JSONL logs and backend can read them.

Do not reset or delete `app_data/taiji.db` during routine verification. Keep local data unless the user explicitly approves a destructive reset or the task is specifically about rebuilding a demo baseline.

## Conventions when extending

- **New business model**: extend `TenantMixin` (`from app.models._tenant_mixin import TenantMixin`) so it gets `tenant_id` and is included in the auto-filter. Otherwise data leaks across tenants. If the model also needs a `db.relationship("Tenant", ...)`, reference the FK by **string** (`foreign_keys="MyModel.tenant_id"`) — not `foreign_keys=[tenant_id]` — because `tenant_id` is injected by `TenantMixin` and the bare name is not bound in the class body at declaration time.
- **New protected endpoint**: stack `@jwt_required()` (and `@require_permission(Permission.XXX)` for permission control), parse with a marshmallow schema via `validate_schema`, delegate to a `services/` function, return via `ok()` / `created()` / `paginated()`. Raise `BusinessError(ErrorCode.XXX)` instead of returning error JSON manually — the global handler shapes it. **Mount new blueprints under `/api/v1/...`** (via `f"{API_V1}/<name>"`); only health-check / monitoring routes live outside the version prefix. **Superuser-only** routes go in `app/api/superadmin.py` with `@superuser_required`.
- **New permission**: add a constant to `app/permissions.py::Permission`, register it in `PERMISSIONS_REGISTRY`, and decide which system roles get it via `SYSTEM_ROLES`. Migration is automatic on the next `flask db upgrade` only if you regenerate one — for runtime addition, update the seed and let admins assign it via the role management UI.
- **New system setting**: add a key to `SETTING_DEFINITIONS` in `app/services/system_setting_service.py` with `description` and `default` value. The `update_settings()` function auto-validates against the definitions dict — unknown keys are rejected with `VALIDATION_ERROR`. Add a seed in a migration if a default DB row is needed.
- **Cross-tenant query** (needed in superadmin paths, public register, etc.): wrap with `with bypass_tenant_filter():` from `app.utils.decorators`. Outside request context (Celery, CLI scripts) the hook auto-skips, but use the context manager anyway for defensive clarity.
- **Celery task that touches DB**: take `tenant_id` as an argument, wrap DB work in `with celery.flask_app.app_context():`, and ideally also `with bypass_tenant_filter():` to make the cross-tenant access explicit (since worker isn't in request context, the hook wouldn't fire anyway — but be explicit).
- **New Celery task**: define in `app/tasks/`, import it in `celery_app.py` so the Worker auto-registers it.
- **New rate-limited endpoint**: import the shared `limiter` from `app` and use `@limiter.limit(...)`. Only carve out a separate `Limiter` instance if the route has fundamentally different limiting policy (as auth does). Both the shared `limiter` and `auth_limiter` use `_get_client_ip()` (reads `X-Forwarded-For` / `X-Real-IP` for reverse proxy support) — do not revert to `get_remote_address`.
- **SSRF-safe HTTP requests**: any endpoint that makes outbound HTTP requests to user-provided URLs must use `safe_requests_get()` from `app/utils/ssrf.py` and validate the URL with `validate_url()` at both submit time (schema) and execution time (worker).
- **Frontend API call**: add a module under `src/api/`, import the default `request` from `./request`, do not bypass the interceptors.
- **Frontend styling**: use `var(--taiji-*)` / `var(--el-*)` variables — never hard-code `#409eff`-style colors, dark mode will break. Section headers use `.page-title` + `.page-icon` pattern (see `TaskManagement.vue`). Page roots use `max-width: 1200px; margin: 0 auto;` to keep wide-screen content centered.
- **Frontend reactivity with polling/active tasks**: when pushing plain objects into `ref([])` arrays and later mutating them in callbacks (timers, polling), always access the item through the reactive proxy (`array.value[idx]` or a helper like `_getActiveTask(id)`) — **never** hold a reference to the raw object from the closure, as mutations on it bypass Vue's reactivity and won't trigger re-renders.
- **Frontend icons**: `<el-icon><IconName /></el-icon>` directly — icons are globally registered, no import needed. Reach for `@element-plus/icons-vue` names rather than ad-hoc SVGs.
- **Frontend permission gates**: use `const { has } = usePermission()` from `composables/usePermission.js`; `v-if="has('user:write')"` on UI elements. Router guards in `router/index.js` use `meta.requiresPermission`. For superuser-only UI / routes, use `v-if="authStore.isSuperuser"` and `meta.requiresSuperuser: true` — RBAC permissions and the superuser flag are independent dimensions.
- **Login flow**: login accepts only global `username` + `password` and enters the first active membership. Users switch tenants from the header `TenantSwitcher`; do not add tenant selection back to the login form.
- **Token revocation on mutations**: any service function that changes a user's role, membership status, active state, or password must set `user.tokens_revoked_at = _revoke_marker()` to invalidate all existing tokens for that user. This ensures stale JWT claims (especially `perms`) are refreshed on next login.
- **Audit logging on mutations**: superadmin / user-management / membership service functions should call `record_audit_log(action=..., resource_type=..., resource_id=..., resource_name=..., before_data=..., after_data=...)` immediately before `db.session.commit()` so the audit row commits with the business write. Use action-name conventions already in the codebase (`user.create` / `user.update` / `user.disable` / `user.enable` / `user.delete` / `password.change` / `superuser.grant` / `superuser.revoke` / `tenant_member.add` / `tenant_member.remove`).
